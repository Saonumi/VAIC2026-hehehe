3. Mode contract và các invariant bắt buộc
3.1 Ask Regulations
Là multi-turn RAG.
Được dùng chat history của đúng conversation hiện tại để resolve đại từ và câu hỏi tiếp nối.
Chỉ dùng nguồn pháp lý APPROVED + ACTIVE, đúng query date hoặc assessment date.
Attachment tùy chọn chỉ là conversation context; không được coi là authority source.
Khi tạo New Chat, memory bắt đầu rỗng.
Không được truy xuất attachment hoặc messages từ conversation khác.
3.2 Document Review
Khi bật mode, hệ thống yêu cầu upload file và chọn assessment date trước khi chạy.
File là REVIEW_TARGET, không phải legal evidence và không vào global KB.
Initial evaluation phải chạy như một tác vụ độc lập, không sử dụng lịch sử chat làm kiến thức.
Mỗi lần chạy tạo một immutable Review Run.
Sau khi có kết quả, người dùng được chat tiếp để hỏi giải thích hoặc đề xuất sửa đổi.
Explainer chỉ được dùng Review Run JSON và evidence đã khóa của run đó.
Muốn đổi file, ngày đánh giá, phạm vi hoặc knowledge snapshot thì phải tạo Review Run mới.
3.3 Add Regulatory Source
Giữ nguyên workflow hiện tại.
File upload ban đầu là UNVERIFIED_AUTHORITY_CANDIDATE.
Chỉ APPROVED + ACTIVE mới được dùng trong retrieval hoặc review evidence.
Chat attachment và review target không được đi qua activation workflow này trừ khi người dùng chủ động chọn Add Regulatory Source và có Human Review.
Invariant quan trọng nhất
Conversation content có thể hỗ trợ hiểu ý người dùng, nhưng không bao giờ tự động trở thành tri thức pháp lý. Legal truth chỉ đến từ nguồn APPROVED + ACTIVE trong snapshot được chỉ định.

4.3 Snapshot rule
Mỗi Review Run phải lưu knowledge_snapshot_id. Nếu sau đó có nguồn pháp lý mới được activate, kết quả cũ vẫn giữ nguyên. Việc đánh giá lại phải tạo Review Run mới với snapshot mới.
5. Review Run bất biến và chat giải thích
UPLOAD FILE    ↓CREATE REVIEW RUN    ↓FREEZE assessment_date + knowledge_snapshot + parser/prompt versions    ↓EXTRACT CLAIMS    ↓RETRIEVE APPROVED EVIDENCE    ↓STRUCTURED EVALUATION    ↓VERIFY OUTPUT    ↓LOCK REVIEW RESULT    ↓OPTIONAL MULTI-TURN EXPLAINER CHAT
5.1 Tại sao không dùng single-turn hoàn toàn?
Single-turn hoàn toàn làm giảm giá trị tương tác. Thiết kế đúng là initial evaluation có tính single-run và tái lập, sau đó cho phép multi-turn explanation trong phạm vi đã khóa. Người dùng vẫn có thể hỏi “vì sao?”, “nguồn nào?”, “đề xuất viết lại?”, nhưng không thể làm thay đổi kết luận gốc chỉ bằng lời nói.
5.2 Review Run state machine
CREATED→ FILE_UPLOADED→ TEXT_EXTRACTED→ CLAIMS_EXTRACTED→ EVIDENCE_RETRIEVED→ ASSESSED→ VERIFIED→ READY→ HUMAN_REVIEWED (optional)Failure states:NEEDS_INPUT | INSUFFICIENT_EVIDENCE | FAILED | CANCELLED
5.3 Re-run rule
Đổi assessment date → Review Run mới.
Đổi review scope → Review Run mới.
Upload phiên bản file khác → Review Run mới.
Knowledge snapshot thay đổi → Review Run mới.
Chỉ hỏi giải thích hoặc yêu cầu draft sửa → vẫn dùng Review Run cũ.

8. Batch Document Review
Nguyên tắc bắt buộc
Một file = một Review Run độc lập. Nhiều Review Run = một Batch Review. Không nối toàn bộ file thành một prompt và không dùng báo cáo A làm evidence cho báo cáo B.
8.1 Batch flow
Upload N files→ Create BatchReview BR-001→ Freeze shared knowledge snapshot and configuration→ Create N independent ReviewRuns→ Queue + bounded worker pool→ Aggregate verified findings→ Generate Batch Review Report→ Enable chat by scope: entire batch / one report / selected findings
8.2 Shared và isolated data
Shared trong batch
Bắt buộc cô lập theo file
assessment_date
review target text
knowledge_snapshot_id
extracted claims
prompt/parser version
retrieved evidence set
review profile/scope
claim assessments and citations
queue configuration
error/retry state
8.3 Batch output
Progress theo từng file: queued, processing, completed, failed, cancelled.
Summary: total documents, total claims, status distribution.
Recurring issue groups: cùng provision, cùng outdated value hoặc cùng missing obligation.
Drill-down từ issue group → affected documents → exact claim → evidence.
Retry/cancel/re-run riêng từng file.
Re-run entire batch with latest approved snapshot tạo batch mới, không ghi đè batch cũ.

9.3 Invariants cần test
File A upload trong Conversation A không được xuất hiện trong Conversation B.
Review target A không được xuất hiện trong Ask Regulations như nguồn pháp lý.
Message trong chat không được index vào global OpenSearch/Neo4j.
Delete conversation phải xóa hoặc expire local attachment index theo retention policy.
New chat không reuse hidden context từ chat trước.

10.2 Review Evaluator rules
Không dùng chat history.
Chỉ dùng target document và approved evidence được truyền vào.
Trả JSON hợp schema; không trả văn xuôi tự do.
Không tạo citation ngoài allowlist.
Khi evidence không đủ, dùng MISSING_EVIDENCE hoặc AMBIGUOUS.
Suggested revision phải dựa trên evidence; nếu không đủ phải để null và requires_human_review=true.
10.3 Review Explainer rules
Chỉ dùng locked Review Run JSON và evidence của run.
Không được thay đổi status hoặc evidence set.
Có thể giải thích, tóm tắt, soạn đề xuất sửa và trả lời follow-up.
Nếu yêu cầu tạo đánh giá với parameter mới, trả action suggestion CREATE_NEW_REVIEW_RUN.

12.3 API invariants
Server derives owner_id from auth token; không nhận owner_id tự do từ client.
Mọi endpoint đọc conversation/review phải verify ownership/role.
Message endpoint phải verify mode contract.
Review result không được update trực tiếp; re-run tạo entity mới.
